Chapter 1What is a Raspberry Pi?

Imagine a full computer — the kind you'd browse the web on, write documents, or even play games — but shrunk down to the size of a credit card. That's a Raspberry Pi. It's a tiny, affordable, single-board computer that can do almost anything a regular PC can do, just in a much smaller package.

Simba's brain runs on a Raspberry Pi 4 Model B with 4 GB of RAM. Here's what makes it special:

  • Quad-core ARM CPU — four processor cores running at 1.5 GHz, enough to run AI models, process voice commands, and control motors all at once.
  • 4 GB RAM — enough memory to hold Simba's AI brain (~750 MB), camera frames, and all the background processes simultaneously.
  • 40 GPIO pins — these are the physical pins on the board that connect to servos, motors, sensors, and microphones. They're Simba's nervous system.
  • CSI camera port — a dedicated connector for the camera ribbon cable (that's how Simba sees).
  • USB, WiFi, Bluetooth — for connecting accessories, remote access, and the web dashboard.

Think of the Pi like a tiny brain sitting inside Simba's body. It reads sensors, thinks about what to do (using AI), and sends commands to motors and servos — all in real time. When you open simba/main.py, you're looking at the very first code that runs when this little computer boots Simba to life:

python -m simba.main
💡 Fun Fact: The Raspberry Pi was originally designed to teach kids programming in schools. Today it powers everything from weather stations to robots like Simba!

Chapter 2What is an Operating System?

An operating system (OS) is the software that manages everything on a computer. It's like the manager of a restaurant — it doesn't cook the food or serve the customers, but it makes sure the cooks have ingredients, the waiters know which tables to serve, and everything runs smoothly.

Simba runs Raspberry Pi OS, which is based on Linux. Linux is free, lightweight, and extremely popular for robotics and servers. Unlike Windows, Linux gives you direct access to hardware — which is exactly what a robot needs.

The most important tool in Linux is the terminal (also called the command line or shell). Instead of clicking icons, you type commands. For example:

# Start Simba
bash scripts/start_simba.sh

# Or start with systemd (runs on boot automatically)
sudo systemctl start simba

The terminal might look intimidating at first, but it's actually faster and more powerful than clicking around. Think of it like texting vs. calling — both work, but texting (the terminal) lets you be precise and efficient.

When Simba boots up, the OS loads first, then it starts the Python program that brings Simba to life. The OS handles the WiFi connection (so you can access the web dashboard), manages memory, and makes sure all of Simba's processes play nicely together.

💡 Tip: You can access Simba's Pi remotely over WiFi using SSH: ssh pi@raspberrypi. No monitor or keyboard needed!

Chapter 3What is Python?

Python is the programming language Simba is written in. If the Raspberry Pi is Simba's body, Python is the language we use to write Simba's thoughts — every decision, every movement, every word Simba understands.

Why Python? Because it reads almost like English. Look at this real code from simba/motion/chassis.py:

def forward(self, speed=None):
    """move forward."""
    speed = speed or self.cruise_speed
    self.set_speed(speed, speed)
    log_event("motion", f"chassis forward at {speed}%")

Even if you've never programmed before, you can probably guess what that does — it makes the robot move forward! Python's simplicity means you spend less time fighting the language and more time building cool things.

To run Python code on the Pi, you open the terminal and type:

# Run Simba's main program
python -m simba.main

# Or run it with flags
python -m simba.main --no-web
python -m simba.main --web-only

Every .py file in the project is a Python file. The Simba project has about 30 of them, organized into folders like simba/motion/, simba/vision/, and simba/voice/. Each folder handles a different part of the robot.

💡 Tip: Python uses indentation (spaces at the start of lines) to organize code into blocks. If something is indented under a def line, it belongs to that function. Neat, right?

Chapter 4What is a Library/Package?

Imagine you're building a house. You could mine your own iron, smelt it into nails, grow trees, mill them into lumber… or you could just buy nails and lumber from a hardware store. Libraries (also called packages) are the hardware store of programming — pre-built code written by other people that you can use in your own project.

Simba uses many libraries. You can see them all listed in requirements_pi.txt:

# hardware control

# voice

sounddevice==0.5.1    # captures audio from the microphone

To install all these libraries at once, you run one command:

pip install -r requirements_pi.txt

pip is Python's package manager — think of it as the delivery service that fetches libraries from the internet and installs them on your Pi.

In the code, you use a library by importing it. Look at the top of simba/motion/imu.py:

from smbus2 import SMBus

This single line gives Simba the ability to talk to the MPU6050 gyroscope sensor. Without this library, you'd have to write hundreds of lines of low-level communication code yourself!

💡 Tip: There are two requirements files — requirements_pi.txt for the Raspberry Pi and requirements_trainer.txt for the training PC. Different machines need different tools!

Chapter 5What is GPIO?

GPIO stands for General Purpose Input/Output. These are the 40 metal pins sticking up from the Raspberry Pi board. They're like Simba's nervous system — electrical signals flow through them to control servos, read sensors, and drive motors.

Each pin can be set to either input (reading data from sensors) or output (sending commands to motors). The Raspberry Pi 4B has 26 usable GPIO pins, plus power (3.3V and 5V) and ground pins.

There are two ways to number the pins, which often confuses beginners:

  • BCM numbering — refers to the Broadcom chip's internal numbers (GPIO 4, GPIO 17, etc.). This is what Simba uses.
  • Board numbering — refers to the physical position on the board (pin 7, pin 11, etc.).

Check out the pin assignments from config/simba_config.yaml:

pins:
  finger_1: 4        # BCM GPIO 4  → physical pin 7
  finger_2: 17       # BCM GPIO 17 → physical pin 11
  finger_3: 27       # BCM GPIO 27 → physical pin 13
  arm_rotation: 22   # BCM GPIO 22 → physical pin 15
  arm_elbow: 23      # BCM GPIO 23 → physical pin 16
  arm_wrist: 24      # BCM GPIO 24 → physical pin 18
  motor_a_in1: 5     # BCM GPIO 5  → physical pin 29
  motor_a_enable: 12 # BCM GPIO 12 → physical pin 32 (PWM)

In the code, Simba writes to these pins to make things move. Here's a snippet from simba/motion/chassis.py that makes the left motor spin forward:

# Set left motor direction to forward
self.pi.write(self.in1, 1)   # IN1 = HIGH
self.pi.write(self.in2, 0)   # IN2 = LOW
⚠️ Warning: Never connect a GPIO pin directly to a motor or high-voltage device! GPIO pins output only 3.3V at a few milliamps. That's why we use motor drivers like the L298N (Chapter 12).

Chapter 6What is PWM?

PWM stands for Pulse Width Modulation. It's a clever trick to control things like servo angles and motor speeds using simple on/off signals. Instead of sending a variable voltage (which GPIO can't do), we rapidly switch a pin between HIGH and LOW — and the ratio of on-time to off-time determines the effective power.

Think of it like a light switch you flip on and off really fast. If it's on 50% of the time and off 50%, the light appears half as bright. If it's on 90% of the time, it's almost fully bright.

In Simba, PWM is used for two things:

  • Servo control — the pulse width (500–2500 μs) determines the angle (0°–180°). See simba/motion/arm.py:
def _angle_to_pulse(self, angle):
    """convert angle (0-180) to pulse width (500-2500 microseconds)."""
    angle = max(0, min(180, angle))
    return int(self.pulse_min + (angle / 180.0) * (self.pulse_max - self.pulse_min))
  • Motor speed — the duty cycle (0–255) controls how fast the wheels spin. From simba/motion/chassis.py:
duty = int(abs(speed) * 2.55)  # 0-100 → 0-255
self.pi.set_PWM_dutycycle(self.ena, min(duty, 255))

Simba uses the pigpio library for PWM because it generates hardware-timed signals, which prevents the jittering you'd get with software-only PWM.

Chapter 7What is I2C?

I2C (pronounced "eye-squared-see") stands for Inter-Integrated Circuit. It's a communication protocol that lets the Raspberry Pi talk to sensors using just two wires: SDA (data) and SCL (clock). Think of it like a two-lane road — one lane carries the actual data, and the other lane keeps a steady beat so both sides know when to send and when to listen.

Each device on the I2C bus has a unique address. Simba's MPU6050 gyroscope uses address 0x68. The Pi (the "master") says, "Hey device at 0x68, send me your accelerometer data!" and the sensor responds.

Here's how Simba reads data from the MPU6050 in simba/motion/imu.py:

from smbus2 import SMBus

# Open I2C bus 1, talk to device at address 0x68
self.bus = SMBus(1)
self.bus.write_byte_data(self.address, self.PWR_MGMT_1, 0)  # wake up!

# Read a 16-bit value from a register
high = self.bus.read_byte_data(self.address, register)
low = self.bus.read_byte_data(self.address, register + 1)
value = (high << 8) | low

The pin assignments from simba_config.yaml confirm the wiring:

i2c_sda: 2    # GPIO 2 — physical pin 3
i2c_scl: 3    # GPIO 3 — physical pin 5
💡 Fun Fact: I2C can connect up to 127 devices on the same two wires! Each device just needs a different address. Simba only uses one (the MPU6050), but you could add more sensors without needing extra wires.

Chapter 8What is I2S?

I2S (pronounced "eye-two-ess") stands for Inter-IC Sound. While I2C is for sensors, I2S is specifically designed for audio. It's the protocol Simba's INMP441 microphone uses to send digital audio data to the Raspberry Pi.

I2S uses three wires:

  • BCLK (Bit Clock) — the metronome that keeps everything in sync, like a drummer keeping the beat.
  • LRCLK (Left/Right Clock) — tells the mic whether to send the left or right audio channel (Simba uses mono, so this just toggles each sample).
  • DIN (Data In) — the actual audio data flows here, one bit at a time, synced to the clock.

The pin mapping from simba_config.yaml:

i2s_bclk: 18     # GPIO 18 — pin 12
i2s_lrclk: 19    # GPIO 19 — pin 35
i2s_din: 20      # GPIO 20 — pin 38

The audio from the INMP441 is captured by the sounddevice library and fed to the Vosk speech recognizer. You can see this in simba/voice/listener.py, where audio arrives at 16,000 samples per second:

self.sample_rate: int = voice_cfg.get("sample_rate", 16000)
self.channels: int = voice_cfg.get("channels", 1)  # mono
💡 Why I2S instead of analog? Digital audio (I2S) is immune to electrical noise. An analog mic near buzzing motors would pick up interference, but the INMP441 sends clean 1s and 0s.

Chapter 9What is CSI?

CSI stands for Camera Serial Interface. It's a dedicated connector on the Raspberry Pi designed specifically for cameras. Instead of using USB (which would be slower and use more CPU), the CSI port provides a direct, high-speed link between the camera sensor and the Pi's GPU.

The camera connects via a thin, flat ribbon cable — a flexible strip of wires that plugs into a small white clip on the Pi board. It might look fragile, but it carries high-speed image data at up to 1080p30 or 640×480 at higher framerates.

Simba uses a 5MP CSI camera module to see the world. The camera captures frames at 640×480 resolution at 15 FPS (frames per second). In simba/vision/camera.py, the picamera2 library controls the camera:

from picamera2 import Picamera2

self.camera = Picamera2()
config = self.camera.create_still_configuration(
    main={"size": self.resolution, "format": "rgb888"},
    buffer_count=2,
)
self.camera.configure(config)
self.camera.start()

Once the camera is running, Simba captures frames for two purposes: the vision pipeline (object detection at 224×224) and the web dashboard (MJPEG streaming at 320×240).

💡 Tip: The ribbon cable has a correct orientation — the blue tape side faces the Ethernet/USB ports. If your camera isn't detected, the cable might be upside down!

Chapter 10What is a Servo Motor?

A servo motor is a motor that can move to a specific angle and hold that position. Unlike regular motors that just spin, a servo says, "I'll rotate to exactly 90 degrees and stay there." This precision makes servos perfect for robot arms, fingers, and wrists.

Simba uses 7 SG90 servo motors: four for the arm (Rotation, Elbow 1, Elbow 2, Wrist) and three for the fingers. Each SG90 can rotate from 0° to 180°.

The secret is in the pulse width. Every 20 milliseconds (50 Hz), you send a pulse. A 500-microsecond pulse means "go to 0°." A 2500-microsecond pulse means "go to 180°." Anything in between maps proportionally.

Here's the actual conversion from simba/motion/arm.py:

def _angle_to_pulse(self, angle):
    """convert angle (0-180) to pulse width (500-2500 microseconds)."""
    angle = max(0, min(180, angle))
    return int(500 + (angle / 180.0) * (2500 - 500))

So 90° = 500 + (90/180) × 2000 = 1500 μs. The pigpio library sends these precisely-timed pulses so the servos don't jitter.

Simba's arm also does smooth interpolation — instead of jumping instantly to an angle, it glides there step by step at 50 updates per second. That's what _move_smooth() does: it creates natural, lifelike motion.

⚠️ Warning: SG90 servos are powered by the 18650 batteries (3.7V), not the Pi's GPIO pins. GPIO pins can only signal the servo; they can't provide the current needed to actually turn the motor.

Chapter 11What is a DC Motor?

A DC motor (Direct Current motor) is the simplest kind of motor — give it power and it spins. Reverse the power and it spins the other way. Unlike servos that move to specific angles, DC motors spin continuously, which makes them perfect for wheels.

Simba has two 5V 100RPM DC motors — one for the left wheel and one for the right. By controlling the speed and direction of each wheel independently, Simba can go forward, backward, turn, or even spin in place (called differential drive).

Speed control is done with PWM (Chapter 6). A higher duty cycle = faster spinning:

# From simba/motion/chassis.py
def set_speed(self, left, right):
    """set raw motor speeds. -100 to 100 for each wheel."""
    left = max(-self.max_speed, min(self.max_speed, left))
    right = max(-self.max_speed, min(self.max_speed, right))

Direction control uses an H-bridge (Chapter 12). By setting two input pins, you choose which way current flows through the motor:

# Forward: IN1=HIGH, IN2=LOW
self.pi.write(self.in1, 1)
self.pi.write(self.in2, 0)

# Backward: IN1=LOW, IN2=HIGH
self.pi.write(self.in1, 0)
self.pi.write(self.in2, 1)

# Stop: both LOW
self.pi.write(self.in1, 0)
self.pi.write(self.in2, 0)

The clever part is differential drive — to turn left, the right wheel spins faster than the left. To spin in place, the wheels turn in opposite directions:

def spin_left(self, speed=None):
    """spin in place to the left."""
    speed = speed or self.turn_speed
    self.set_speed(-speed, speed)  # left backward, right forward!
💡 Fun Fact: Simba's third "wheel" is actually a ball caster — a small ball in a socket that rolls freely in any direction, providing front support without fighting the turns.

Chapter 12What is the L298N Motor Driver?

The Raspberry Pi's GPIO pins can output only 3.3V at a few milliamps — barely enough to light an LED. A DC motor needs hundreds of milliamps at 5V or more. If you connected a motor directly to GPIO, you'd fry the Pi! The L298N is the middleman that solves this problem.

The L298N is an H-bridge integrated circuit. It takes tiny signals from the Pi and uses them to switch big power from the battery to the motors. Think of it like a relay — you flip a tiny switch to control a big one.

Simba's wiring from simba_config.yaml:

# Motor A (left wheel)          # Motor B (right wheel)
motor_a_in1: 5    # direction    motor_b_in3: 13   # direction
motor_a_in2: 6    # direction    motor_b_in4: 16   # direction
motor_a_enable: 12 # PWM speed   motor_b_enable: 26 # PWM speed

How it works:

  • IN1 & IN2 — control Motor A's direction. One HIGH, one LOW = forward. Swap them = backward.
  • EN1 (Enable) — PWM signal that controls speed. 0% = stopped, 100% = full speed.
  • Same pattern for Motor B with IN3, IN4, and EN2.

The chassis code in simba/motion/chassis.py sets the PWM frequency to 1000 Hz for smooth speed control:

self.pi.set_PWM_frequency(self.ena, 1000)
self.pi.set_PWM_frequency(self.enb, 1000)
⚠️ Important: The L298N needs two power sources — Vss (pin 8) for the chip's logic (3.3V from Pi) and Vs (pin 16) for the motors (from the batteries). Don't mix them up!

Chapter 13What is the MPU6050?

The MPU6050 is a tiny chip that can sense motion and orientation. It's a 6-axis sensor, meaning it contains two sensors in one:

  • Accelerometer (3 axes) — measures acceleration in X, Y, and Z directions. When sitting still, it reads Earth's gravity (1g downward), which tells you which way is "up."
  • Gyroscope (3 axes) — measures rotational speed. If you twist the sensor, the gyro tells you how fast and in which direction.

In Simba, the MPU6050 is mounted on the hand. It tells Simba whether the hand is level, tilted left, tilted right, pointing up, or pointing down — crucial information when trying to grab objects!

The sensor talks to the Pi via I2C (Chapter 7) at address 0x68. Here's how simba/motion/imu.py reads the accelerometer:

def read_accel(self):
    """read accelerometer values in g."""
    ax = self._read_raw(self.ACCEL_XOUT_H) / self.accel_scale
    ay = self._read_raw(self.ACCEL_XOUT_H + 2) / self.accel_scale
    az = self._read_raw(self.ACCEL_XOUT_H + 4) / self.accel_scale
    return (ax, ay, az)

And here's the clever orientation detection:

def get_hand_orientation(self):
    """determine hand orientation from accelerometer."""
    ax, ay, az = self.read_accel()
    if abs(az) > 0.8 and abs(ax) < 0.3 and abs(ay) < 0.3:
        return "level"
    elif ay > 0.5:
        return "tilted_right"
    elif ay < -0.5:
        return "tilted_left"

Before using it, the MPU6050 needs calibration — it takes 100 readings at rest and averages them to find the zero point. This compensates for tiny manufacturing imperfections in the chip.

💡 Tip: The MPU6050 also has a built-in temperature sensor! Simba reads it with: (raw / 340.0) + 36.53. It's not super accurate, but it's a nice bonus.

Chapter 14What is the INMP441?

The INMP441 is a tiny digital MEMS microphone. MEMS stands for Micro-Electro-Mechanical Systems — it's a microphone built into a chip smaller than your fingernail. Unlike the analog mic in your phone's headset, the INMP441 outputs digital audio directly via I2S (Chapter 8).

This is Simba's ear. It picks up your voice commands — "get me the screwdriver," "hello," "I love you" — and sends the audio as a stream of numbers to the Raspberry Pi.

Key specs for Simba's setup:

  • Sample rate: 16,000 Hz — meaning 16,000 audio snapshots per second. That's enough for clear speech recognition but not overkill (music uses 44,100 Hz).
  • Channels: 1 (mono) — Simba only needs one ear.
  • Chunk size: 4,000 samples — audio is processed in batches of 4,000 samples (0.25 seconds each).

From simba/voice/listener.py, the audio stream is set up like this:

self._stream = sd.InputStream(
    samplerate=self.sample_rate,   # 16000 Hz
    channels=self.channels,         # 1 (mono)
    dtype="float32",
    blocksize=self.chunk_size,      # 4000 samples
    callback=self._audio_callback,  # called every chunk
)

Each chunk of audio is converted from float32 to int16 bytes and fed to the Vosk speech recognizer:

# Convert float32 → int16 bytes for Vosk
audio_bytes = (indata[:, 0] * 32767).astype(np.int16).tobytes()

if self.recognizer.AcceptWaveform(audio_bytes):
    result = json.loads(self.recognizer.Result())
    text = result.get("text", "").strip().lower()

When Vosk recognizes a complete sentence, it fires a callback that sends the text to Simba's brain for processing — all happening in real time!

💡 Why digital? The INMP441 is perfect for a noisy robot. Because it outputs digital audio over I2S, the signal is immune to electrical interference from motors and servos. An analog mic would pick up buzzing and humming.

Chapter 15What are 18650 Batteries?

18650 batteries are rechargeable lithium-ion cells. The name comes from their dimensions: 18mm diameter × 65mm long × 0 (cylindrical). They look like slightly chunkier AA batteries and are the same type used in laptop battery packs and Tesla cars.

Each 18650 cell provides:

  • Nominal voltage: 3.7V (ranges from 3.0V empty to 4.2V fully charged)
  • Capacity: ~2200mAh — meaning it can deliver 2,200 milliamps for one hour, or 220mA for ten hours

Simba uses 2× 18650 cells in parallel to power the servos and DC motors. Here's the difference between series and parallel:

Parallel keeps the voltage at 3.7V (safe for the servos and motors) while doubling the capacity. More capacity = Simba runs longer between charges.

The simba_config.yaml includes battery thresholds:

charging:
  low_battery_threshold: 3.3   # volts — time to charge!
  full_battery_threshold: 4.1  # volts — fully charged

Meanwhile, the Raspberry Pi itself is powered by a separate power bank via USB-C. This keeps the Pi's power supply clean and stable, isolated from the noisy motor power.

⚠️ Safety: Lithium batteries can be dangerous if short-circuited, overcharged, or punctured. Always use a protection circuit like the TP4056 (next chapter) and never expose them to heat or water.

Chapter 16What is the TP4056?

The TP4056 is a tiny, inexpensive module that safely charges lithium-ion batteries. It's about the size of a postage stamp and takes care of everything — charging, monitoring, and protecting the 18650 batteries from damage.

Here's what it does:

  • Charging — plug in a Micro USB cable and it charges the 18650 at a safe 1A rate. An LED turns red while charging and blue/green when full.
  • Overcharge protection — automatically stops charging at 4.2V so the battery doesn't swell or catch fire.
  • Over-discharge protection — cuts off power if the battery drops below ~2.5V, preventing permanent damage to the cells.
  • Short-circuit protection — if something goes wrong, it disconnects instantly.

The key feature of the version with protection (usually marked "TP4056 with DW01A") is the OUT+ and OUT- pads. These are the protected outputs — they go to Simba's servos and motors. The BAT+ and BAT- connect directly to the 18650 cells.

When Simba detects low battery (below 3.3V as set in the config), the brain can automatically navigate to the charging pad:

def _handle_charge(self):
    """navigate to charging pad."""
    self.state.transition("charging")
    self._thinking_text = "looking for charging pad..."
💡 Tip: You can charge Simba while it's running — the TP4056 handles balancing charge current and load current simultaneously. Just plug in the USB cable!

Chapter 17What is AI?

Artificial Intelligence (AI) is not magic, and it's not a robot uprising waiting to happen. At its core, AI is simply software that can recognize patterns and make decisions — things that previously required a human brain.

Think about how you recognize a screwdriver. You've seen hundreds of screwdrivers in your life — different sizes, colors, brands — and your brain learned the pattern: "long handle, metal shaft, flat or cross-shaped tip." AI does the same thing, but with math instead of neurons.

Simba uses AI in several ways:

  • Object Recognition — the camera sees a frame of pixels, and AI identifies "that's a screwdriver" with 85% confidence. (See simba/vision/detector.py)
  • Speech Recognition — the microphone captures sound waves, and AI converts them to text: "get me the screwdriver." (See simba/voice/listener.py)
  • Natural Language Understanding — a small language model (Qwen 2.5, 500M parameters) gives Simba personality and reasoning. (See simba/core/brain.py)
  • Speaker Verification — AI recognizes whose voice is speaking, so Simba only obeys its owner.

Here's the AI lineup from Simba's config:

ai:
  llm_model_path: "models/qwen2.5-0.5b-instruct-q4_k_m.gguf"
  vision_model_path: "models/mobilenetv2_feature_extractor.tflite"
  confidence_threshold: 0.6  # 60% sure = good enough

Chapter 18What is Machine Learning?

Machine Learning (ML) is the process of teaching a computer by showing it examples rather than writing explicit rules. Instead of coding "if the handle is red and the shaft is 15cm, it's a screwdriver," you show the computer 1,000 photos of screwdrivers and 1,000 photos of wrenches, and it figures out the differences on its own.

There are two distinct phases:

  1. Training — happens on a powerful PC. You feed thousands of labeled examples to an algorithm, and it learns patterns. This is the slow, compute-heavy part.
  2. Inference — happens on the Raspberry Pi. The trained model looks at new data and makes predictions. This is fast and lightweight.

Simba's vision training happens in trainer/train_vision.py. Here's the real workflow:

Chapter 19What is a Neural Network?

A neural network is a type of machine learning model inspired by the human brain. It's made up of layers of artificial "neurons" connected to each other — each neuron takes in numbers, does a little math, and passes the result to the next layer.

Simba uses MobileNetV2 — a neural network with about 3.4 million parameters, designed to run efficiently on mobile and embedded devices. It takes a 224×224 pixel image and compresses it into a 1,280-dimensional feature vector — a list of 1,280 numbers that describe the image.

From simba/vision/detector.py:

def extract_features(self, frame):
    """extract a 1280-dim feature vector from a frame using mobilenetv2."""
    resized = self._resize_frame(frame, self.input_size)  # → 224×224
    input_data = np.expand_dims(resized.astype(np.float32) / 255.0, axis=0)
    self.interpreter.set_tensor(self._input_index, input_data)
    self.interpreter.invoke()
    output = self.interpreter.get_tensor(self._output_index)
    return output.flatten()  # → 1280 numbers

How does it learn? During training, the network starts with random weights (the numbers that connect neurons). It sees an image, makes a guess, checks if it's wrong, and adjusts the weights slightly. After seeing thousands of images, those weights converge into patterns that correctly identify objects.

Simba also uses a second neural network — Qwen 2.5 (500 million parameters, quantized to 4-bit) — for language understanding. This is the "brain" that gives Simba its personality and conversational ability.

💡 Key Insight: The 1,280 numbers from MobileNetV2 aren't random — they represent meaningful features like edges, textures, shapes, and objects. The SVM classifier (Chapter 18) then uses these features to make the final decision.

Chapter 20What is Computer Vision?

Computer vision is the field of AI that teaches computers to "see" — to look at images and understand what's in them. For humans, seeing is effortless. But for a computer, an image is just a grid of numbers (pixels), and figuring out that a particular pattern of numbers is a "screwdriver" is incredibly hard.

Here's how Simba sees the world, step by step:

  1. Capture — the CSI camera grabs a 640×480 image (307,200 pixels, each with Red, Green, and Blue values).
  2. Resize — the image is shrunk to 224×224 pixels for the neural network.
  3. Feature Extraction — MobileNetV2 converts the image into 1,280 meaningful numbers.
  4. Classification — the SVM classifier maps those numbers to a label: "screwdriver," "wrench," etc.

From simba/vision/detector.py:

def classify(self, frame):
    """classify a frame: extract features then run svm prediction."""
    features = self.extract_features(frame)       # → 1280 numbers
    features_2d = features.reshape(1, -1)
    label = self.classifier.predict(features_2d)[0]  # → "screwdriver"
    probas = self.classifier.predict_proba(features_2d)[0]
    confidence = float(np.max(probas))               # → 0.87
    return (str(label), confidence)

Simba also does sliding window detection — it scans the image at multiple scales, checking each region for objects:

How does Simba tell a wrench from a screwdriver? It doesn't use rules like "wrenches are U-shaped." Instead, MobileNetV2 learned to extract features — edge patterns, textures, shapes — from thousands of training images. The SVM then learned which combination of features means "wrench" vs. "screwdriver."

When Simba scans an area, it physically rotates its arm to sweep the camera across the room, classifying objects at each angle and remembering where they are. Then when you say "get me the screwdriver," it knows exactly where to look!

Chapter 21What is Speech Recognition?

Speech recognition is the art of turning your voice into text a computer can read. When you say "get me the bottle," Simba needs to convert those sound waves into the string "get me the bottle" before it can understand what you want.

Chapter 22What is an LLM?

An LLM — Large Language Model — is an AI that has read billions of words and learned to predict what comes next. When you give it a sentence, it generates a response word-by-word, choosing the most likely continuation each time. Think of it as autocomplete on steroids.

Simba uses Qwen 2.5 (0.5B parameters) — a model with 500 million "knobs" that were tuned during training. Why 0.5B? Because the Raspberry Pi 4 only has 4 GB of RAM. A model like ChatGPT has hundreds of billions of parameters — way too big. Qwen 2.5–0.5B is the sweet spot: smart enough to hold a conversation, small enough to run on a Pi.

In simba/core/brain.py, the LLM is loaded using ollama:

Simba's personality is defined in a system prompt that tells the LLM: "You are Simba, a friendly bionic arm robot. You speak in short, warm sentences. You are loyal and eager to help." This is what gives Simba its character.

Why "Instruct"? The "instruct" variant is fine-tuned to follow instructions rather than just complete text. Ask it a question, and it answers — rather than just rambling on.

Chapter 23What is Quantization?

Imagine you have a super-detailed photograph. Quantization is like compressing it into a smaller JPEG — you lose a tiny bit of quality, but the file is way smaller. The same idea works for AI models.

Neural network weights are normally stored as 32-bit or 16-bit floating point numbers. Quantization reduces these to 8-bit, 4-bit, or even lower — making the model file drastically smaller and faster to run.

Simba's LLM uses Q4_K_M quantization. Let's decode that name:

Size comparison:
• Full precision (FP32): ~2 GB for 0.5B params
• Q4_K_M (4-bit): ~400 MB — fits in Pi's RAM!

Why do we quantize for Raspberry Pi? The Pi 4 has only 4 GB of RAM shared between the OS, camera, voice recognition, the web server, and everything else. A full-precision model would gobble up all the memory. Q4_K_M gives us about 95% of the quality at 20% of the size.

The model is served via ollama. The prompt string concatenates the system context, the emotion state, and the detected user intent.

Chapter 24What is TFLite?

TensorFlow Lite (TFLite) is Google's framework for running neural networks on small, resource-constrained devices — phones, microcontrollers, and yes, Raspberry Pis. It's like a diet version of TensorFlow: same brain, smaller body.

Regular TensorFlow is designed for beefy GPUs and servers. TFLite strips away everything unnecessary and optimizes for speed on CPUs and edge devices. The models are saved as .tflite files — compact, pre-optimized, and ready to run.

Simba uses TFLite to run MobileNetV2 for vision. Here's how it loads in simba/vision/detector.py:

import tflite_runtime.interpreter as tflite

# Load the model
self.interpreter = tflite.Interpreter(
    model_path="models/mobilenetv2_feature_extractor.tflite"
)
self.interpreter.allocate_tensors()

# Run inference
self.interpreter.set_tensor(self._input_index, input_data)
self.interpreter.invoke()
output = self.interpreter.get_tensor(self._output_index)

Notice we import tflite_runtime — not the full TensorFlow package. The runtime is just ~5 MB instead of ~500 MB. On a Pi with limited storage and RAM, that's a huge win.

The pipeline: Train a big model on a powerful computer → Convert to TFLite → Deploy the tiny .tflite file to the Pi. The Pi only needs to run the model, not train it.

Chapter 25What is MobileNetV2?

MobileNetV2 is a lightweight image recognition model designed by Google specifically for mobile and embedded devices. While big models like ResNet-152 have 60 million parameters, MobileNetV2 has only 3.4 million — yet performs remarkably well.

Its secret? Depthwise separable convolutions — a fancy way of saying "break the math into two smaller, cheaper steps instead of one expensive step." Think of it like washing dishes: instead of scrubbing each dish with a sponge soaked in soap and water simultaneously, you soap everything first, then rinse everything. Same result, less effort.

Simba uses MobileNetV2 as a feature extractor. Instead of asking "is this a cat or a dog?", it asks "describe what you see as a list of 1,280 numbers." These numbers form a feature vector — a numerical fingerprint of the image.

From simba/vision/detector.py:

Those 1,280 numbers capture the essence of the image — shapes, textures, colors — in a format that a simple classifier can understand.

Chapter 26What is an SVM?

An SVM (Support Vector Machine) is one of the simplest yet most powerful classifiers in machine learning. Its job: given a set of numbers (features), decide which category they belong to.

Imagine you have a table full of red and blue marbles. An SVM draws the best possible line between them — the line that maximizes the gap (called the margin) between the two groups. In higher dimensions, this "line" becomes a hyperplane.

Why SVM instead of a big neural network? Because Simba's MobileNetV2 already does the hard work of understanding the image. The SVM just needs to sort the 1,280-number summary into categories like "bottle," "ball," or "cup." It's fast, uses almost no memory, and trains on just a handful of examples.

In simba/vision/detector.py:

Chapter 27What is MFCC?

MFCC stands for Mel-Frequency Cepstral Coefficients. That's a mouthful, so let's break it down: MFCCs are a way to turn the sound of your voice into a compact set of numbers — a voice fingerprint.

Here's the intuition: your voice has unique qualities — pitch, tone, the shape of your vocal tract. MFCCs capture these qualities by analyzing which frequencies are present in your speech, weighted by how human ears actually perceive sound (the Mel scale).

Simba extracts 13 MFCC coefficients plus their deltas (rate of change), giving 26 numbers per audio frame. This is done in simba/voice/speaker_verify.py:

# Extract voice fingerprint
mfccs = librosa.feature.mfcc(
    y=audio_float,
    sr=self.sample_rate,     # 16000 Hz
    n_mfcc=13,               # 13 coefficients
    n_fft=512,
    hop_length=160,
    n_mels=40,
)
deltas = librosa.feature.delta(mfccs)
features = np.vstack([mfccs, deltas]).T   # shape: (n_frames, 26)

These features feed into a Gaussian Mixture Model (GMM) that learns what your voice "looks like" in number-space. When Simba hears someone speak, it extracts their MFCCs and asks: "Does this sound like my owner?" If the confidence is above 0.65 (the threshold), it accepts the command.

Chapter 37Wiring the INMP441 Microphone

The INMP441 is a high-quality digital MEMS microphone that uses the I2S (Inter-IC Sound) protocol. Unlike cheap analog mics, it outputs clean digital audio directly — no ADC (analog-to-digital converter) needed.

What each wire does:

  • SCK (BCLK) — Bit Clock: the Pi tells the mic when to send each bit
  • WS (LRCLK) — Word Select: toggles between left and right channels (we use mono)
  • SD (DOUT) — Serial Data: the actual audio data flows through here
  • L/R — Tied to GND to select the left channel
Audio settings from simba_config.yaml:

Chapter 38Connecting the CSI Camera

The CSI camera (Camera Serial Interface) connects directly to the Raspberry Pi via a flat ribbon cable — no USB needed. This gives Simba low-latency, high-quality video for object detection.

Step-by-step connection:

  1. Power off the Pi completely (not just shutdown — unplug power)
  2. Locate the CSI port — it's the small connector between the HDMI ports and the audio jack
  3. Gently pull up the black plastic latch on the CSI connector
  4. Slide the ribbon cable in with the blue/silver side facing the Ethernet port and the gold contacts facing the HDMI ports
  5. Push the latch back down to lock the cable
  6. Connect the other end to the camera module the same way

Chapter 47arm.py — rotate(), raise_arm(), wrist()

Each of these three functions moves one specific servo. They all follow the same pattern: clamp the angle to safe limits, acquire the thread lock, smoothly move to the target, then log what happened.

First, the magic formula that converts an angle to a servo pulse width:

def _angle_to_pulse(self, angle):
    """convert angle (0-180) to pulse width (500-2500 microseconds)."""
    angle = max(0, min(180, angle))
    return int(self.pulse_min + (angle / 180.0) * (self.pulse_max - self.pulse_min))

Now the three movement functions:

def rotate(self, angle):
    """rotate arm on y-axis."""
    angle = max(self.rotation_min, min(self.rotation_max, angle))
    with self._lock:
        self._moving = True
        self.current["rotation"] = self._move_smooth(
            self.rotation_pin, self.current["rotation"], angle
        )
        self._moving = False
    log_event("motion", f"arm rotated to {angle}°")

def raise_arm(self, angle):
    """move arm elbow up/down. Higher angle = arm raised higher."""
    angle = max(self.elbow_min, min(self.elbow_max, angle))
    with self._lock:
        self.current["elbow"] = self._move_smooth(
            self.elbow_pin, self.current["elbow"], angle
        )

def wrist(self, angle):
    """move wrist up/down."""
    angle = max(self.wrist_min, min(self.wrist_max, angle))
    with self._lock:
        self.current["wrist"] = self._move_smooth(
            self.wrist_pin, self.current["wrist"], angle
        )
💡 Safety clamping: Notice max(min, min(max, angle)) at the top of each function? This ensures you can never tell the arm to go beyond its physical limits — preventing stripped gears and broken parts.

Chapter 48arm.py — move_smooth()

If you told a servo to instantly jump from 0° to 180°, it would slam into position violently — bad for the servo, bad for whatever Simba is holding. The _move_smooth() method is the solution: it interpolates between the current angle and the target, stepping gradually.

Here's the single-servo version (used internally by rotate(), raise_arm(), wrist()):

def _move_smooth(self, pin, current_angle, target_angle, speed=None):
    """smoothly interpolate servo from current to target angle."""
    if speed is None:
        speed = self.move_speed    # default: 1.5 deg/step

    step = speed if target_angle > current_angle else -speed
    angle = current_angle

    while abs(angle - target_angle) > speed:
        angle += step
        self._set_servo(pin, angle)
        time.sleep(0.02)  # 50hz update rate — 20ms per step

    # snap to final position
    self._set_servo(pin, target_angle)
    return target_angle

And here's the multi-servo version that moves all four servos simultaneously — so the arm traces a smooth arc instead of moving one joint at a time:

def move_smooth(self, target_angles, speed=None):
    """move all arm servos simultaneously to target angles."""
    # Calculate how many steps needed (based on largest movement)
    steps = max(
        abs(targets["rotation"] - self.current["rotation"]),
        abs(targets["elbow"] - self.current["elbow"]),
        abs(targets["wrist"] - self.current["wrist"]),
    )
    num_steps = max(1, int(steps / speed))

    for i in range(1, num_steps + 1):
        t = i / num_steps     # linear interpolation factor: 0→1
        for key, pin in [("rotation", self.rotation_pin),
                         ("elbow", self.elbow_pin),
                         ("wrist", self.wrist_pin)]:
            angle = self.current[key] + t * (targets[key] - self.current[key])
            self._set_servo(pin, angle)
        time.sleep(0.02)
💡 Why 0.02 seconds? That's a 50 Hz update rate — matching the standard servo PWM frequency. Moving faster would waste CPU. Moving slower would look jerky. 50 Hz is the sweet spot.

Chapter 51hand.py — grab() and release()

These two functions are the core of Simba's gripping ability. The key insight: fingers close gradually, not all at once. This prevents crushing delicate objects and allows the grip to conform around irregular shapes.

def grab(self):
    """close all fingers gradually to grab an object."""
    log_event("motion", "hand grabbing!")
    with self._lock:
        current_angles = list(self.finger_angles)
        while any(a < self.closed_angle for a in current_angles):
            for i in range(3):
                if current_angles[i] < self.closed_angle:
                    current_angles[i] = min(
                        current_angles[i] + self.grab_speed,  # +2°
                        self.closed_angle                      # max 150°
                    )
                    self._set_finger(i, current_angles[i])
            time.sleep(0.02)   # 50hz — same as servo refresh
        self._grip_state = "closed"

The release() function is the mirror image — fingers open gradually:

def release(self):
    """open all fingers gradually to release object."""
    log_event("motion", "hand releasing!")
    with self._lock:
        current_angles = list(self.finger_angles)
        while any(a > self.open_angle for a in current_angles):
            for i in range(3):
                if current_angles[i] > self.open_angle:
                    current_angles[i] = max(
                        current_angles[i] - self.grab_speed,  # -2°
                        self.open_angle                        # min 30°
                    )
                    self._set_finger(i, current_angles[i])
            time.sleep(0.02)
        self._grip_state = "open"
⚠️ Why not just snap closed? Imagine grabbing an egg. Instant closure = scrambled egg. At 2° per step with 20ms delays, it takes about 1.2 seconds to close fully — gentle enough for delicate objects but fast enough to feel responsive.

Chapter 53chassis.py — ChassisController Overview

The ChassisController in simba/motion/chassis.py drives Simba's wheels using differential drive — the same principle behind tanks, wheelchairs, and Roombas. Two motorized wheels plus a ball caster for balance.

The controller uses an L298N motor driver — a chip that takes low-power GPIO signals and converts them into high-power motor control. Each motor needs 3 pins:

class ChassisController:
    """2wd differential drive with l298n motor driver."""

    def __init__(self, config):
        pins = config["pins"]
        self.in1 = pins["motor_a_in1"]       # gpio 5  ┐
        self.in2 = pins["motor_a_in2"]       # gpio 6  ├─ left motor
        self.ena = pins["motor_a_enable"]    # gpio 12 ┘ (speed via PWM)
        self.in3 = pins["motor_b_in3"]       # gpio 13 ┐
        self.in4 = pins["motor_b_in4"]       # gpio 16 ├─ right motor
        self.enb = pins["motor_b_enable"]    # gpio 26 ┘ (speed via PWM)

        motor_cfg = config["motors"]
        self.max_speed = motor_cfg["max_speed"]       # 100
        self.cruise_speed = motor_cfg["cruise_speed"] # 60
        self.turn_speed = motor_cfg["turn_speed"]     # 45

        # set PWM frequency for enable pins
        self.pi.set_PWM_frequency(self.ena, 1000)  # 1 kHz
        self.pi.set_PWM_frequency(self.enb, 1000)
💡 Why L298N? The Raspberry Pi's GPIO can only output ~3.3V at a few milliamps — nowhere near enough to spin a motor. The L298N acts as a power amplifier, letting tiny GPIO signals control motors that draw hundreds of milliamps from a separate power supply.

Chapter 54chassis.py — forward(), backward(), turn_left(), turn_right()

These four functions are Simba's basic movement vocabulary. They all work by calling set_speed(left, right) with different values for each wheel:

def forward(self, speed=None):
    """move forward."""
    speed = speed or self.cruise_speed     # default: 60%
    self.set_speed(speed, speed)           # both wheels same speed
    log_event("motion", f"chassis forward at {speed}%")

def backward(self, speed=None):
    """move backward."""
    speed = speed or self.cruise_speed
    self.set_speed(-speed, -speed)         # negative = reverse

def turn_left(self, speed=None):
    """turn left (right wheel faster)."""
    speed = speed or self.turn_speed       # default: 45%
    self.set_speed(speed * 0.3, speed)     # left slow, right fast

def turn_right(self, speed=None):
    """turn right (left wheel faster)."""
    speed = speed or self.turn_speed
    self.set_speed(speed, speed * 0.3)     # left fast, right slow

Under the hood, each motor direction is set by the IN pins on the L298N:

def _set_motor_a(self, speed):
    """set left motor speed. positive = forward, negative = backward."""
    if speed > 0:
        self.pi.write(self.in1, 1)   # IN1=HIGH, IN2=LOW → forward
        self.pi.write(self.in2, 0)
    elif speed < 0:
        self.pi.write(self.in1, 0)   # IN1=LOW, IN2=HIGH → backward
        self.pi.write(self.in2, 1)
    else:
        self.pi.write(self.in1, 0)   # both LOW → brake
        self.pi.write(self.in2, 0)
    duty = int(abs(speed) * 2.55)    # 0-100% → 0-255 duty cycle
    self.pi.set_PWM_dutycycle(self.ena, min(duty, 255))
💡 Turning math: For turn_left(), the left wheel runs at 30% of the right wheel's speed. This creates an arc turn rather than a sharp pivot. For spinning in place, Simba has spin_left() which runs wheels in opposite directions.

Chapter 55chassis.py — PWM Speed Control

How do you make a motor go at half speed when you can only send HIGH (on) or LOW (off)? The answer is PWM — Pulse Width Modulation. Instead of giving the motor steady power, you flick it on and off thousands of times per second. The percentage of time it's ON determines the effective speed.

The set_speed() function handles the conversion from a friendly 0–100 percentage to the 0–255 PWM duty cycle that pigpio expects:

def set_speed(self, left, right):
    """set raw motor speeds.
    
    args:
        left: -100 to 100 (negative = backward)
        right: -100 to 100
    """
    left = max(-self.max_speed, min(self.max_speed, left))
    right = max(-self.max_speed, min(self.max_speed, right))
    with self._lock:
        self._set_motor_a(left)
        self._set_motor_b(right)
        self._current_speed = (left, right)

The duty cycle conversion formula:

# Inside _set_motor_a() and _set_motor_b():
duty = int(abs(speed) * 2.55)    # 0-100 → 0-255
self.pi.set_PWM_dutycycle(self.ena, min(duty, 255))
💡 Why 1000 Hz? The PWM frequency (set_PWM_frequency(self.ena, 1000)) determines how fast the on/off switching happens. At 1000 Hz (1 kHz), the motor "sees" a smooth average voltage. Too slow and the motor would visibly stutter. Too fast and the driver chip can't keep up.

Chapter 56imu.py — IMUReader Overview

The IMU (Inertial Measurement Unit) is Simba's sense of balance and orientation. The IMUReader in simba/motion/imu.py talks to an MPU6050 chip — a tiny sensor that measures acceleration and rotation — mounted on Simba's hand.

Communication happens over I2C (a two-wire serial protocol) using the smbus2 library. The sensor lives at address 0x68 and stores its readings in numbered registers:

class IMUReader:
    """reads mpu6050 accelerometer + gyroscope data over i2c."""

    # MPU6050 register addresses
    PWR_MGMT_1 = 0x6b      # power management
    ACCEL_XOUT_H = 0x3b    # accelerometer data starts here
    GYRO_XOUT_H = 0x43     # gyroscope data starts here

    def __init__(self, config):
        imu_cfg = config["imu"]
        self.address = imu_cfg["i2c_address"]    # 0x68
        self.accel_range = imu_cfg["accel_range"] # ±2g
        self.gyro_range = imu_cfg["gyro_range"]   # ±250 deg/s
        self.sample_rate = imu_cfg["sample_rate"] # 50 Hz

        # Scale factors convert raw 16-bit integers to real units
        accel_scales = {2: 16384.0, 4: 8192.0, 8: 4096.0, 16: 2048.0}
        self.accel_scale = accel_scales.get(self.accel_range, 16384.0)

        # Initialize sensor
        self.bus = SMBus(1)
        self.bus.write_byte_data(self.address, self.PWR_MGMT_1, 0)  # wake up!

Reading a 16-bit value from two 8-bit registers:

def _read_raw(self, register):
    """read 16-bit signed value from register pair."""
    high = self.bus.read_byte_data(self.address, register)
    low = self.bus.read_byte_data(self.address, register + 1)
    value = (high << 8) | low       # combine two bytes
    if value >= 0x8000:
        value -= 0x10000            # convert to signed
    return value
💡 Calibration is crucial! At startup, calibrate() takes 100 readings while the sensor is still, calculates the average "at rest" values, and subtracts them as offsets. Without calibration, even a level surface might read as tilted.

Chapter 57imu.py — get_hand_orientation()

This function is how Simba knows whether its hand is level, tilted, or pointing up/down. It reads the accelerometer (which always measures gravity) and translates the raw X/Y/Z values into human-readable orientation strings.

def get_hand_orientation(self):
    """determine hand orientation from accelerometer.
    
    returns:
        str: 'level', 'tilted_left', 'tilted_right',
             'pointing_up', 'pointing_down'
    """
    ax, ay, az = self.read_accel()

    if abs(ay) > 0.8 and abs(ax) < 0.5 and abs(az) < 0.5:
        return "level"          # Hand is level, gravity mostly pulling Y down
    elif ax < -0.5:
        return "tilted_right"   # Gravity pulling X negative (Right tilt)
    elif ax > 0.5:
        return "tilted_left"    # Gravity pulling X positive (Left tilt)
    elif az > 0.5:
        return "pointing_up"    # Gravity pulling Z positive (Hand up)
    elif az < -0.5:
        return "pointing_down"  # Gravity pulling Z negative (Hand down)
    return "level"

The complementary function get_tilt_angle() gives a precise angle from vertical:

def get_tilt_angle(self):

    ax, ay, az = self.read_accel()

        math.sqrt(ax * ax + ay * ay), az

def _capture_loop(self) -> None:

Chapter 59camera.py — capture_for_inference()

When Simba's AI brain needs to identify an object, it doesn't use the full 640×480 camera frame. Instead, capture_for_inference() resizes it to 224×224 pixels — the exact size that MobileNetV2 expects.

def capture_for_inference(
    self,
    size: Tuple[int, int] = (224, 224),
) -> Optional[np.ndarray]:
    """capture and resize a frame for mobilenetv2 inference.
    
    args:
        size: target (width, height) — default 224×224.
    returns:
        numpy array of shape (224, 224, 3), dtype uint8
    """
    frame = self.capture_frame()    # get latest frame (640×480)
    if frame is None:
        return None

    if Image is not None:
        # Pillow resize (high quality)
        img = Image.fromarray(frame)
        img = img.resize(size, Image.BILINEAR)
        return np.array(img)

    # Fallback: numpy nearest-neighbour resize
    h, w = frame.shape[:2]
    target_h, target_w = size[1], size[0]
    row_idx = (np.arange(target_h) * h // target_h).astype(int)
    col_idx = (np.arange(target_w) * w // target_w).astype(int)
    return frame[np.ix_(row_idx, col_idx)]

Why exactly 224×224? MobileNetV2 was trained on ImageNet images resized to 224×224. Neural networks are rigid about their input dimensions — feed it a different size and it will crash or give garbage results. It's like trying to plug a USB-A cable into a USB-C port — the connector has to match exactly.

💡 Dual fallback: Notice the code tries Pillow first (bilinear filtering for quality), then falls back to a pure numpy nearest-neighbour resize if Pillow isn't installed. Simba always has a Plan B!

Chapter 60camera.py — get_mjpeg_frame()

When you open Simba's web dashboard and see a live camera feed, you're watching an MJPEG stream — a continuous series of JPEG images sent to your browser. The get_mjpeg_frame() function handles the pipeline from raw camera frame to compressed JPEG bytes.

def get_mjpeg_frame(self) -> Optional[bytes]:
    """encode the latest frame as a jpeg byte string for web streaming."""
    frame = self.capture_frame()       # Step 1: get RGB numpy array
    if frame is None:
        return None

    if Image is not None:
        img = Image.fromarray(frame)
        # Step 2: resize to stream resolution (320×240)
        img = img.resize(self.stream_resolution, Image.BILINEAR)
        # Step 3: compress to JPEG bytes
        buf = io.BytesIO()
        img.save(buf, format="JPEG", quality=self.jpeg_quality)
        return buf.getvalue()          # raw JPEG bytes

Chapter 61detector.py — ObjectDetector Overview

How does Simba know what it's looking at? That's the job of ObjectDetector in simba/vision/detector.py. It uses a clever two-stage pipeline — think of it like a factory assembly line with two stations.

Stage 1: Feature Extraction — A MobileNetV2 neural network (running via TFLite) looks at a camera frame and distills it down to a vector of 1,280 numbers. These numbers are like a "fingerprint" of the image — they capture what's in the picture without being the actual pixels.

Stage 2: Classification — A Support Vector Machine (SVM) takes those 1,280 numbers and says "that's a bottle" or "that's a cup" with a confidence score.

💡 Why two stages? Running a full neural network classifier on a Raspberry Pi would be too slow. By splitting the work — a lightweight TFLite model for features and a fast SVM for classification — Simba gets real-time performance on limited hardware.

Chapter 62detector.py — extract_features()

This is where the magic starts. The extract_features() method takes a raw camera frame — any size — and turns it into a compact fingerprint of 1,280 numbers. Think of it like summarizing a book into a single paragraph.

def extract_features(self, frame: np.ndarray) -> np.ndarray:
    """extract a 1280-dim feature vector from a frame using mobilenetv2."""
    if self.interpreter is None:
        return np.zeros(self.feature_dim, dtype=np.float32)

    # resize to model input size
    resized = self._resize_frame(frame, self.input_size)

    # prepare input tensor: (1, h, w, 3) float32 normalised to [0, 1]
    input_data = np.expand_dims(resized.astype(np.float32) / 255.0, axis=0)

    # check expected dtype from model
    input_details = self.interpreter.get_input_details()
    expected_dtype = input_details[0]["dtype"]
    if expected_dtype == np.uint8:
        input_data = np.expand_dims(resized.astype(np.uint8), axis=0)

    self.interpreter.set_tensor(self._input_index, input_data)
    self.interpreter.invoke()

    output = self.interpreter.get_tensor(self._output_index)
    features = output.flatten().astype(np.float32)

    return features

Here's the step-by-step flow:

💡 Pixel normalization: Neural networks work best with small numbers. Dividing by 255 converts pixel values (0–255) to the range 0.0–1.0. Some models prefer uint8 instead — the code checks and adapts automatically.

Chapter 63detector.py — classify()

Once we have the 1,280-number fingerprint, the SVM classifier takes over. It's like a librarian who can look at a book summary and instantly tell you which shelf it belongs on.

def classify(self, frame: np.ndarray) -> Tuple[str, float]:
    """classify a frame: extract features then run svm prediction."""
    if self.classifier is None:
        return ("unknown", 0.0)

    features = self.extract_features(frame)
    if np.all(features == 0):
        return ("unknown", 0.0)

    try:
        features_2d = features.reshape(1, -1)

        label = self.classifier.predict(features_2d)[0]

        # get probability if the classifier supports it
        confidence = 0.0
        if hasattr(self.classifier, "predict_proba"):
            probas = self.classifier.predict_proba(features_2d)[0]
            confidence = float(np.max(probas))
        elif hasattr(self.classifier, "decision_function"):
            decision = self.classifier.decision_function(features_2d)
            confidence = float(
                1.0 / (1.0 + np.exp(-np.max(np.abs(decision))))
            )

        return (str(label), confidence)

    except Exception as exc:
        logger.error("classification error: %s", exc)
        return ("unknown", 0.0)

The method tries two ways to get a confidence score:

  1. predict_proba() — Direct probability from SVM (best option)
  2. decision_function() — Raw score converted to pseudo-probability via sigmoid: 1 / (1 + e^(-x))
⚠️ Zero features mean trouble: If extract_features() returns all zeros (model not loaded), classify() bails out immediately with ("unknown", 0.0) rather than feeding garbage to the SVM.

Chapter 64detector.py — detect_objects() with Sliding Window

What if there are multiple objects in the frame? A single classify() call would only see the whole image. The sliding window approach breaks the image into overlapping tiles and classifies each one independently — like scanning a page with a magnifying glass.

def detect_objects(self, frame: np.ndarray) -> List[Dict]:
    """detect objects using a sliding window approach."""
    h, w = frame.shape[:2]
    detections: List[Dict] = []

    for scale in self._window_scales:       # [1.0, 0.5]
        win_w = int(w * scale)
        win_h = int(h * scale)

        if win_w < 32 or win_h < 32:
            continue

        stride_x = max(1, int(win_w * self._window_stride))
        stride_y = max(1, int(win_h * self._window_stride))

        for y in range(0, h - win_h + 1, stride_y):
            for x in range(0, w - win_w + 1, stride_x):
                tile = frame[y:y + win_h, x:x + win_w]
                label, conf = self.classify(tile)

                if label != "unknown" and conf >= self.confidence_threshold:
                    detections.append({
                        "label": label,
                        "confidence": round(conf, 4),
                        "bbox": [x, y, win_w, win_h],
                    })

    # simple non-maximum suppression
    detections = self._nms(detections)
    return detections

After scanning, Non-Maximum Suppression (NMS) cleans up duplicate detections — if two overlapping tiles both see a "bottle", only the highest-confidence one survives. Each detection includes a bounding box [x, y, width, height] so Simba knows where the object is in the frame.

def search_for_object(self, target_label: str) -> Tuple[bool, int, float]:
    """scan all angles looking for a specific object."""
    logger.info("searching for '%s' across %d positions",
                target_label, len(self.angles))

Chapter 68listener.py — The Audio Callback

The _audio_callback() method is the heartbeat of voice recognition. It's called automatically by sounddevice every time a new chunk of audio arrives from the microphone — roughly every 250 milliseconds.

def _audio_callback(self, indata, frames: int, time_info, status) -> None:
    """sounddevice input stream callback — process incoming audio."""
    if status:
        logger.debug("audio stream status: %s", status)

    if self.recognizer is None:
        return

    # convert float32 numpy array to int16 bytes for vosk
    try:
        import numpy as np
        audio_bytes = (indata[:, 0] * 32767).astype(np.int16).tobytes()
    except Exception:
        audio_bytes = indata.tobytes()

    try:
        if self.recognizer.AcceptWaveform(audio_bytes):
            result = json.loads(self.recognizer.Result())
            text = result.get("text", "").strip().lower()
            if text:
                self._dispatch_text(text)
        else:
            # check partial results for real-time feedback
            partial = json.loads(self.recognizer.PartialResult())
            partial_text = partial.get("partial", "").strip().lower()
            if partial_text:
                logger.debug("partial: %s", partial_text)
    except Exception as exc:
        logger.error("recognizer error: %s", exc)

The critical conversion step: indata[:, 0] * 32767. The microphone gives us float32 values between -1.0 and 1.0. Vosk needs int16 values between -32,768 and 32,767. Multiplying by 32,767 bridges that gap.

Chapter 70command_parser.py — CommandParser Overview

When someone says "hey simba, could you grab the water bottle please?" — the robot needs to understand: action = fetch, target = water bottle. That's the job of CommandParser in simba/voice/command_parser.py.

class CommandParser:
    """extracts structured commands from recognized speech text."""

    def parse(self, text: str) -> Optional[Dict[str, Optional[str]]]:
        """parse recognized text into a structured command."""
        normalized = text.strip().lower()

        # remove common filler words for better matching
        cleaned = self._clean_text(normalized)

        # try exact substring matching first (faster)
        result = self._try_exact_match(cleaned, normalized)
        if result is not None:
            return result

        # fall back to fuzzy matching
        result = self._try_fuzzy_match(cleaned, normalized)
        if result is not None:
            return result

        return None

# Output: {"action": "fetch", "target": "bottle", "modifier": None}

Before matching, filler words are stripped out — "um," "uh," "like," "please," "could you," "simba" — so the matching only sees the important words. This makes the parser much more forgiving of natural speech patterns.

💡 Two-phase matching: Exact match runs first (fast O(n) substring check). Only if nothing is found does it fall back to the slower fuzzy matching. Speed matters when you want the robot to respond quickly!

Chapter 71command_parser.py — Exact vs Fuzzy Matching

Exact matching is straightforward — it checks if any known command phrase appears as a substring in the input:

def _try_exact_match(self, cleaned, original):
    for patterns, action, has_target in _command_patterns:
        for pattern in patterns:
            if pattern in cleaned or pattern in original:
                target = None
                if has_target:
                    target = self._extract_target(cleaned, pattern)
                return {"action": action, "target": target,
                        "modifier": None}
    return None

Fuzzy matching kicks in when speech recognition produces imperfect text. It uses Python's SequenceMatcher to calculate similarity, with a 70% threshold:

_fuzzy_threshold = 0.70

def _similarity(a: str, b: str) -> float:
    return SequenceMatcher(None, a, b).ratio()

def _try_fuzzy_match(self, cleaned, original):
    words = cleaned.split()

    for patterns, action, has_target in _command_patterns:
        for pattern in patterns:
            pattern_words = pattern.split()
            pattern_len = len(pattern_words)

            # slide a window of pattern length over the input
            for i in range(max(1, len(words) - pattern_len + 1)):
                window = " ".join(words[i:i + pattern_len])
                score = _similarity(window, pattern)

                if score > best_score and score >= _fuzzy_threshold:
                    best_score = score
                    best_action = action

So if Vosk mishears "grab the" as "crab the" — the similarity is about 73%, which exceeds the 70% threshold. The command still gets recognized! This sliding window approach means the fuzzy matcher looks at every possible phrase-length segment in the input.

The feature extraction pipeline creates a 26-dimensional voiceprint per audio frame:

def _extract_features(self, audio: np.ndarray):
    """extract mfcc features and deltas from an audio chunk."""
    # 13 MFCC coefficients
    mfccs = librosa.feature.mfcc(
        y=audio_float, sr=self.sample_rate,
        n_mfcc=13, n_fft=512, hop_length=160, n_mels=40,
    )

    # compute first-order deltas (rate of change)
    deltas = librosa.feature.delta(mfccs)

    # stack: shape (26, n_frames) -> (n_frames, 26)
    features = np.vstack([mfccs, deltas]).T
    return features
💡 What are MFCCs? Mel-Frequency Cepstral Coefficients capture the shape of your vocal tract — your unique "voice shape." The deltas add information about how your voice changes over time. Together, they're like a detailed voice fingerprint.

Chapter 79memory.py — remember_object() & find_object()

These two methods are the core of Simba's memory — storing objects and retrieving them later.

Remembering an object:

def remember_object(self, label, position_angle, confidence,
                    embedding=None):
    """store or update an object in memory."""
    label = label.strip().lower()
    now = datetime.now().isoformat()

    with self._lock:
        if label in self._objects:
            entry = self._objects[label]
            entry["position_angle"] = position_angle
            entry["confidence"] = confidence
            entry["last_seen_timestamp"] = now
            entry["times_seen"] = entry.get("times_seen", 0) + 1
            if embedding is not None:
                entry["embedding"] = embedding
        else:
            # check capacity
            if len(self._objects) >= self.max_items:
                self._evict_oldest()

            self._objects[label] = {
                "label": label,
                "position_angle": position_angle,
                "confidence": confidence,
                "last_seen_timestamp": now,
                "first_seen_timestamp": now,
                "times_seen": 1,
                "embedding": embedding,
            }

        self._dirty = True

    self.save()

If the object already exists, the entry is updated (new position, new timestamp, incremented counter). If it's new and memory is full, the oldest object is evicted first.

Finding an object:

def find_object(self, label: str) -> Optional[Dict[str, Any]]:
    """find an object in memory by label."""
    label = label.strip().lower()

    with self._lock:
        entry = self._objects.get(label)
        if entry is not None:
            return dict(entry)  # return a copy
    return None
💡 Safety copy: Notice dict(entry)? It returns a copy of the object data, not a reference. This prevents external code from accidentally modifying the internal memory database.

Chapter 80memory.py — Eviction & Capacity

Simba's memory isn't infinite — it's capped at 500 items by default. When memory is full and a new object arrives, the LRU (Least Recently Used) eviction strategy kicks in: the object that hasn't been seen for the longest time gets forgotten.

def _evict_oldest(self) -> None:
    """remove the oldest (least recently seen) object to make room."""
    if not self._objects:
        return

    oldest_label = min(
        self._objects,
        key=lambda k: self._objects[k].get("last_seen_timestamp", ""),
    )
    logger.info("evicting oldest object '%s' to make room", oldest_label)
    del self._objects[oldest_label]

The save mechanism uses atomic file writes to prevent data corruption — even if the robot loses power mid-save:

def save(self) -> bool:
    """persist memory to the json file."""
    with self._lock:
        if not self._dirty:
            return True

        # write atomically via temp file
        tmp_path = self.db_path + ".tmp"
        with open(tmp_path, "w", encoding="utf-8") as fh:
            json.dump(data, fh, indent=2, ensure_ascii=False)
        os.replace(tmp_path, self.db_path)

        self._dirty = False
        return True

The trick: data is written to memory.json.tmp first, then os.replace() atomically swaps it into place. If power is lost during the write, the .tmp file is the one that's corrupted — the original memory.json stays intact.

⚠️ The dirty flag optimization: If nothing has changed (self._dirty = False), save() returns immediately without touching the disk. This prevents unnecessary I/O on the Pi's SD card, extending its lifespan.

def update_from_event(self, event_type: str) -> str:
    mapping = _event_emotion_map.get(event_type)
    if mapping is None:
        return self._emotion

    new_emotion, default_intensity = mapping

    with self._lock:
        current_emotion = self._emotion
        current_intensity = self._intensity

    # if same emotion, boost intensity
    if current_emotion == new_emotion:
        intensity = min(1.0, current_intensity + 0.2)
    else:
        intensity = default_intensity

    self.set_emotion(new_emotion, intensity)
    return new_emotion

Chapter 84emotions.py — decay()

Just like real moods, Simba's emotions don't last forever. The decay() method gradually fades the current emotion back toward neutral — like a ball rolling slowly downhill.

def decay(self) -> None:
    """slowly decay the current emotion toward neutral."""
    with self._lock:
        if self._emotion == "neutral":
            return

        elapsed = time.time() - self._last_update
        decay_amount = self.decay_rate * (elapsed / self.update_interval)

        self._intensity -= decay_amount
        self._intensity = max(0.0, self._intensity)

        if self._intensity <= 0.05:
            old = self._emotion
            self._emotion = "neutral"
            self._intensity = 0.5
            self._last_update = time.time()
            logger.debug("emotion decayed: %s -> neutral", old)

Chapter 92train_vision.py — How Object Training Works

How does Simba learn to recognize a wrench vs. a hammer? The answer: MobileNetV2 + SVM — a proven technique called transfer learning. Here's the pipeline:

Step 1: Each image is resized to 224×224 and normalized to [-1, 1]:

def _preprocess_image(self, image_path: str) -> np.ndarray:
    img = Image.open(image_path).convert("RGB")
    img = img.resize(self.input_shape, Image.LANCZOS)
    arr = np.array(img, dtype=np.float32)
    # mobilenetv2 normalization: [0, 255] -> [-1, 1]
    arr = (arr / 127.5) - 1.0
    arr = np.expand_dims(arr, axis=0)
    return arr

Step 2: MobileNetV2 extracts a 1280-dimensional feature vector from each image. Step 3: A LinearSVC with probability calibration learns to classify:

base_svc = LinearSVC(
    C=1.0,
    max_iter=max_iter,
    class_weight="balanced",
    random_state=42,
)
model = CalibratedClassifierCV(base_svc, cv=min(cv_folds, len(x_train)))
model.fit(x_train, y_train)
💡 Why Not Train MobileNet? Training a full neural network requires a GPU and thousands of images. By using MobileNetV2 as a frozen feature extractor and only training a lightweight SVM on top, you can train with just 10-20 photos per object on a regular laptop!

Chapter 93train_voice.py — How Voice Training Works

Simba recognizes your voice so he only obeys his owner. The technique: MFCC features + Gaussian Mixture Model (GMM). It's the same approach used in professional speaker verification systems.

Step 1: Extract 13 MFCC coefficients plus their first and second derivatives (39 dimensions total per audio frame):

def extract_mfcc(self, audio_path: str) -> np.ndarray:
    audio, sr = librosa.load(audio_path, sr=self.sample_rate, mono=True)
    mfccs = librosa.feature.mfcc(
        y=audio, sr=sr,
        n_mfcc=self.n_mfcc,   # 13
        n_fft=512,
        hop_length=160,
        n_mels=40,
    )
    delta = librosa.feature.delta(mfccs, order=1)
    delta2 = librosa.feature.delta(mfccs, order=2)
    combined = np.vstack([mfccs, delta, delta2]).T
    return combined

Step 2: Fit a GMM with 16 components on all concatenated frames:

gmm = GaussianMixture(
    n_components=n_comp,
    covariance_type="diag",
    max_iter=200,
    n_init=3,
    random_state=42,
)
gmm.fit(combined_features)

At runtime: New audio is scored against the GMM. If the log-likelihood exceeds a threshold, it's recognized as the owner's voice:

log_likelihood = model.score(features)
is_owner = log_likelihood >= threshold
💡 Minimum Samples: You need at least 3 voice recordings. More is better — try 5-10 clips of you saying different phrases. The GMM needs variety to learn your voice's unique characteristics.

Chapter 99Troubleshooting

Things will go wrong — that's part of building robots! Here are the most common issues and their fixes:

🔧 Servo Jitter / Shaking

  • Cause: Insufficient power to PCA9685, or PWM signal noise
  • Fix: Use a dedicated 5V 3A power supply for the servo board. Add a 470μF capacitor across the power rails. Check that pigpiod is running.

📷 Camera Not Found

  • Cause: Camera not enabled or ribbon cable loose
  • Fix: Check /boot/config.txt has start_x=1 and gpu_mem=128. Reseat the ribbon cable. Test with libcamera-still -o test.jpg

🎤 No Audio Input

  • Cause: INMP441 microphone not detected or I2S not enabled
  • Fix: Verify dtoverlay=googlevoicehat-soundcard in /boot/config.txt. Check arecord -l lists a capture device. Run arecord -d 3 test.wav to test.

🐌 Slow LLM Response

  • Cause: Qwen2.5 uses a lot of RAM and CPU on the Pi
  • Fix: Use the Q4_K_M quantized model (already default). Set n_ctx: 512 in config to limit context. Close the web dashboard while using voice commands.

📶 WiFi Drops

  • Cause: RPi WiFi power management turns off the radio
  • Fix: Disable WiFi power saving: sudo iwconfig wlan0 power off. Make it permanent by adding to /etc/rc.local.

🔋 Battery Low Behavior

  • Cause: Voltage drops below threshold
  • Fix: The low_battery event triggers "sleepy" emotion and Simba transitions to CHARGING. Use at least a 5V 3A battery pack. Check voltage with cat /sys/class/power_supply/*/voltage_now.
💡 Check the Logs: Always check journalctl -u simba -f for live logs, or look in data/logs/. The dashboard's activity log also shows recent events. Most problems leave a clear error message!
